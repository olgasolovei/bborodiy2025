# Відповіді на контрольні запитання до лабораторної роботи №4

---

### **1. Визначте дії, які необхідно виконати для забезпечення перевірки вимог (область знань “Requirements Verification” у SWEBOK)**

Згідно з **SWEBOK (Software Engineering Body of Knowledge)**, перевірка вимог включає такі дії:

1. **Оцінка якості вимог** — перевірка на повноту, несуперечливість, однозначність і здійсненність.
2. **Перевірка відповідності вимог стандартам** — використання критеріїв ISO/IEC/IEEE 29148:2011.
3. **Перевірка трасованості (Traceability)** — встановлення зв’язків між бізнес-цілями, вимогами, тестами та артефактами.
4. **Рецензування документів (Requirements Review)** — колективний перегляд вимог із залученням стейкхолдерів.
5. **Валідація моделей і прототипів** — перевірка через діаграми, сценарії використання, прототипи та юзер-сторі.
6. **Підготовка приймальних критеріїв (Acceptance Criteria)** — визначення об’єктивних умов перевірки кожної вимоги.

---

### **2. Визначте відмінності між термінами перевірка (Verification) і утвердження (Validation) вимог**

| Параметр      | Verification                                                                                           | Validation                                                                                               |
| ------------- | ------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------- |
| **Мета**      | Перевірити, *чи правильно побудована система згідно з вимогами* (“Are we building the system right?”). | Перевірити, *чи створена система задовольняє потреби користувача* (“Are we building the right system?”). |
| **Об’єкт**    | Специфікації, моделі, діаграми, код, документація.                                                     | Реальні сценарії використання, прототипи, тестування.                                                    |
| **Виконавці** | Інженери з якості, аналітики, розробники.                                                              | Замовники, кінцеві користувачі, Product Owner.                                                           |
| **Методи**    | Статичний аналіз, review, аудит, інспекція.                                                            | Прототипування, приймальні тести, юзер-сторі, сценарії Gherkin.                                          |

### **3. Перелік ключових слів мови Gherkin**

Мова **Gherkin** використовується для опису сценаріїв приймального тестування у форматі “Given–When–Then” (BDD — Behavior Driven Development).
Ключові слова:

* **Feature** — описує функціональність системи.
* **Scenario** — конкретний приклад або тестовий сценарій.
* **Given** — початкові умови тесту.
* **When** — дія користувача або системи.
* **Then** — очікуваний результат.
* **And**, **But** — використовуються для продовження умов або дій.
* **Background** — спільні умови для кількох сценаріїв.
* **Scenario Outline / Examples** — шаблони сценаріїв із параметрами.

---

### **4. Назвіть методи обговорення документу з вимогами**

Основні методи обговорення вимог:

1. **Review (Огляд)** — неформальне або формальне читання вимог усіма учасниками.
2. **Walkthrough (Прохід по документу)** — автор пояснює документ групі колег, які ставлять запитання.
3. **Inspection (Інспекція)** — формалізована перевірка з використанням чек-листів і протоколів.
4. **Prototyping (Прототипування)** — демонстрація функціональності у вигляді макету або PlantUML-прототипу.
5. **Model-based discussion** — обговорення через діаграми (UML, Use Case, Sequence).
6. **Workshops (робочі зустрічі)** — колективне уточнення та пріоритезація вимог із замовником і користувачами.
