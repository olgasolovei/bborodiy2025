# Таблиця функціональних вимог (FR)

| №         | Опис функціональної вимоги                                                                                              | Пріоритет | Джерело (зацікавлена особа)    |
| --------- | ----------------------------------------------------------------------------------------------------------------------- | --------- | ------------------------------ |
| **FR1**   | Система повинна здійснювати **збір даних із підключених IoT-пристроїв (сенсорів навантаження крана)** в реальному часі. | Високий   | IoT Engineer, System Architect |
| **FR1.1** | Підтримка зв’язку з **Azure IoT Hub** для прийому телеметрії.                                                           | Високий   | System Architect               |
| **FR1.2** | Дані мають оновлюватися з частотою не менше ніж **1 раз на 5 секунд**.                                                  | Високий   | Site Safety Engineer           |
| **FR1.3** | У випадку втрати з’єднання з IoT-пристроєм система повинна **реєструвати помилку зв’язку**.                             | Середній  | System Administrator           |
| **FR2** | Система повинна **аналізувати отримані дані за допомогою AI-модуля**, щоб визначити перевищення допустимого навантаження. | Високий | ML Engineer, Product Owner |
| **FR2.1** | AI-модуль повинен розраховувати **коефіцієнт ризику перекидання** на основі даних навантаження та історії. | Високий | ML Engineer |
| **FR2.2** | Система повинна автоматично **генерувати попередження** при досягненні критичного рівня ризику. | Високий | Site Safety Engineer |
| **FR2.3** | Попередження повинні бути **категоризовані за рівнем небезпеки** (низький, середній, критичний). | Середній | Safety Expert |
| **FR3** | Система повинна **візуалізувати інформацію про стан кранів** через веб-інтерфейс. | Високий | Site Safety Engineer |
| **FR3.1** | На панелі моніторингу має відображатись **список кранів**, рівень навантаження, статус IoT-зв’язку та попередження. | Високий | Site Safety Engineer |
| **FR3.2** | Користувач повинен мати можливість **фільтрувати дані за об’єктами або періодом часу**. | Середній | Project Manager |
| **FR3.3** | У разі аварійного стану елемент інтерфейсу має **підсвічуватись червоним кольором і дублюватися звуковим сигналом.** | Високий | Crane Operator |
| **FR4** | Система повинна **вести журнал подій (лог)** із фіксацією часу, типу події та відповідального користувача. | Високий | Site Safety Engineer, QA Engineer |
| **FR4.1** | Лог має містити всі дії користувачів (вхід у систему, перегляд, реакція на сповіщення). | Середній | QA Engineer |
| **FR4.2** | Дані журналу повинні зберігатися не менше **30 днів**. | Середній | Project Manager |
| **FR5** | Система повинна надавати **звіти про безпеку** (кількість інцидентів, середній рівень ризику, точність прогнозів AI). | Середній | Project Manager |
| **FR5.1** | Користувач може експортувати звіт у **PDF або Excel форматі**. | Середній | Project Manager |
| **FR5.2** | Звіт має містити **метрики точності роботи AI** (Precision, Recall). | Низький | ML Engineer |
| **FR6** | Система повинна **керувати користувачами та ролями доступу**. | Високий | System Administrator |
| **FR6.1** | Адміністратор може додавати, редагувати та деактивувати користувачів. | Високий | System Administrator |
| **FR6.2** | Система має підтримувати ролі: *Operator, Safety Engineer, Project Manager, Administrator.* | Високий | Product Owner |
| **FR7** | Система повинна забезпечити **сповіщення користувачів про критичні ситуації**. | Високий | Safety Engineer |
| **FR7.1** | Сповіщення мають доставлятися через веб-інтерфейс і електронну пошту. | Середній | Site Safety Engineer |
| **FR7.2** | Користувач повинен мати можливість налаштувати **пороги спрацьовування** сповіщень. | Середній | Site Safety Engineer |
