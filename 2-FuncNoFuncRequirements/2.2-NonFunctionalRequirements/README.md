## Таблиця нефункціональних вимог (NFR) до системи.

| FR                     | NFR       | Опис нефункціональної вимоги                                                                                                 |
| ---------------------- | --------- | ---------------------------------------------------------------------------------------------------------------------------- |
| **FR1**                | **NFR1**  | Система повинна підтримувати **обробку не менше 1000 повідомлень IoT-телеметрії за хвилину** без втрати даних.               |
| **FR1.1**              | **NFR2**  | Канал зв’язку з Azure IoT Hub повинен мати **затримку не більше 2 секунд** між надходженням даних і їхнім записом у базу.    |
| **FR1.3**              | **NFR3**  | У разі втрати з’єднання з IoT-пристроєм система повинна **автоматично відновлювати зв’язок протягом 10 секунд**.             |
| **FR2**                | **NFR4**  | Алгоритми штучного інтелекту повинні забезпечувати **точність визначення перевантаження не нижче 95%**.                      |
| **FR2.1**              | **NFR5**  | AI-модуль повинен завершувати аналіз одного пакета даних **менше ніж за 1 секунду**.                                         |
| **FR2.2**              | **NFR6**  | Система повинна зберігати історію усіх ризикових подій **мінімум за останні 6 місяців**.                                     |
| **FR3**                | **NFR7**  | Інтерфейс користувача повинен бути **зрозумілим, без перевантаження елементами**, час відкриття сторінки ≤ 3 с.              |
| **FR3.1**              | **NFR8**  | Веб-додаток має **підтримувати сучасні браузери (Chrome, Edge, Firefox)**.                                                   |
| **FR3.3**              | **NFR9**  | Звукові та візуальні сповіщення мають **працювати без затримки навіть при високому навантаженні системи.**                   |
| **FR4**                | **NFR10** | Журнал подій має бути **захищений від несанкціонованої зміни**; доступ лише користувачам з роллю “Admin”.                    |
| **FR4.2**              | **NFR11** | Обсяг логів за 30 днів не повинен перевищувати **1 ГБ** на кожен будівельний майданчик.                                      |
| **FR5**                | **NFR12** | Система має генерувати звіти за запитом **менше ніж за 10 секунд**, навіть при великому обсязі даних.                        |
| **FR6**                | **NFR13** | Паролі користувачів повинні зберігатись **у зашифрованому вигляді (SHA-256 або bcrypt)**.                                    |
| **FR6.1**              | **NFR14** | Розмежування прав доступу повинно реалізовуватись через **рольову модель RBAC (Role-Based Access Control)**.                 |
| **FR7**                | **NFR15** | Система повинна мати **коефіцієнт готовності (Availability) не нижче 99,5%** у робочий час.                                  |
| **FR7.1**              | **NFR16** | Сповіщення електронною поштою мають доставлятись **не пізніше ніж за 5 секунд після події.**                                 |
| **FR7.2**              | **NFR17** | Інтерфейс налаштування порогів сповіщення має бути **інтуїтивно зрозумілим і без технічних знань користувача.**              |
| **FR2.3**              | **NFR18** | Класифікація ризиків повинна бути **візуально стандартизована (зелений, жовтий, червоний)** для швидкої реакції користувача. |
| **FR1–FR7 (загальні)** | **NFR19** | Система має бути **масштабованою** — здатною обробляти дані з **10+ будівельних майданчиків одночасно.**                     |
| **FR1–FR7 (загальні)** | **NFR20** | Система повинна відповідати **вимогам безпеки ISO/IEC 27001** і забезпечувати шифрування всіх переданих даних.               |
