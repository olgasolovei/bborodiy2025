# Відповіді на контрольні запитання до лабораторної роботи №3

---

### **1. Які документи відповідно до SWEBOK включені в список рекомендованих для опису комплексних проектів з розробки програмного забезпечення?**

Відповідно до **SWEBOK (Software Engineering Body of Knowledge)**, для опису комплексних проєктів рекомендується використовувати такі документи:

* **Vision Document** — опис загальної концепції, цілей і меж продукту;
* **Software Requirements Specification (SRS)** — специфікація вимог;
* **Software Design Description (SDD)** — опис архітектури та компонентів;
* **Software Project Management Plan (SPMP)** — план управління проєктом;
* **Software Test Plan (STP)** — план тестування;
* **User Manual / Installation Guide** — документація для користувачів і адміністраторів;
* **Change Request Document** — запити на зміну вимог або функцій.

---

### **2. Який стандарт описує структуру документа «специфікація програмних вимог»?**

Структуру документа **«Software Requirements Specification» (SRS)** описує міжнародний стандарт:
**ISO/IEC/IEEE 29148:2011 — Systems and Software Engineering — Life Cycle Processes — Requirements Engineering.**

Цей стандарт визначає вимоги до побудови, змісту, атрибутів якості та відстежуваності вимог.

---

### **3. Які дії необхідно виконувати для «управління змінами» до вимог?**

Процес **управління змінами (Change Management)** включає такі дії:

1. **Ініціація зміни** — подання запиту (Change Request).
2. **Аналіз впливу** — оцінка впливу зміни на систему, бюджет, строки.
3. **Затвердження або відхилення** — рішення керівного комітету чи Product Owner.
4. **Оновлення артефактів** — внесення змін у SRS, діаграми, код або тест-кейси.
5. **Відстеження реалізації** — перевірка виконання зміни у новій версії продукту.

---

### **4. Що означає базова версія вимог?**

**Базова версія вимог (Requirements Baseline)** — це **затверджений і контрольований набір вимог**, який приймається як офіційна основа для подальшої розробки.
Будь-які зміни після затвердження базової версії потребують формальної процедури зміни (Change Control).

---

### **5. Які варіанти документування змін до вимог ви знаєте?**

Існують такі основні варіанти документування змін:

* **Change Log** — журнал змін із зазначенням дати, автора, опису та статусу.
* **Change Request Form (CRF)** — окремий документ або запис у системі відстеження (JIRA, Azure DevOps).
* **Версіонування документів (Version Control)** — створення нових версій SRS із мітками (v1.1, v1.2 тощо).
* **Traceability Matrix** — оновлення матриці відстежуваності для фіксації впливу змін на пов’язані вимоги, модулі чи тести.
